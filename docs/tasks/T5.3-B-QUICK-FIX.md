## 两个根的根因分析

### cn-006-FP-01（幻觉 3%，conf=0.85）

**LLM 输出：** `source_text` 不含数字（文本是"质量保证金应在缺陷责任期满后14天内返还给承包人。承包人应提供质量保证金担保函。"），但 LLM 看到"质量保证金"关键词就推理了 3%。

**根因：** prompt 规则 6 说"禁止推导"，但 LLM 不听。prompt 是建议，不是约束。需要在代码层做 **source_text 校验**：如果 source_text 不含阿拉伯数字也不含中文数字（一二三…），拒绝该条提取。

**修复文件：** `tests/road2_llm_extractor.py:120-131`，在 `parse_extractions()` 的 for 循环里加校验。

### cn-020-POS-01（两条路径都 MISSING）

**链路追踪：**

1. `extract_for_sample()` 的 alias 表 `{"履约担保": "cn-020"}` 正确命中 ✓
2. `run_extracted_path()` 构建 `structured_list` 时用的 `field` 是 `ext.get("field_label")` = `"履约担保"` ✗
3. `kernel.validate()` 里 `_preserve_structured_inputs()` 构建 `rule_label_map_si`，键是规则的 `condition_params.label` = `"履约担保比例"` 
4. `inject_structured_fields()` 做 `rule_lookup.get("履约担保")` → **无匹配** → 整条 structured_input 被丢弃

**问题定位在第 2 步：** `field_label` 来自 LLM 输出（"履约担保"），而 kernel 侧期望的是规则本身的 label（"履约担保比例"）。alias 表解决了 rule_id 映射，但没有解决 field 名字的传递。

**修复文件：** `tests/road2_compare.py:107-154`，在 `run_extracted_path()` 的 `structured_list` 构建处，用 `RULE_ID_TO_LABEL` 把 LLM 输出的 field_label 替换为规则的标准 label。

## 改动清单

### 改 1：`tests/road2_llm_extractor.py` — source_text 数字校验

在 `parse_extractions()` 函数中（第 120-131 行），`results.append(...)` 之前加校验：

```python

source = data.get("source_text", "")
has_chinese_num = any(c in source for c in "一二三四五六七八九十百千万亿")
```
