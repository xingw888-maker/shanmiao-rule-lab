## 背景

路二阶段一黄金数据集基线已建立。规则分发已修复，15 条 numeric_comparison 全部进入运行包。当前正则路径准确率 76.1%（35/46），假阳性 1 条（cn-008），假阴性 2 条（cn-010、cn-020）。

剩余误差来源分三类：
- 正则无法匹配中文数字（"二十四个月"→regex 找不到"24"）
- 正则无法匹配复杂句式（"万分之一"/"万分之三"/"百分之三"→单位映射失败）
- 正则误匹配错误语义上下文（"28天内提交结算资料"≠"28天内组织验收"）

T5.2 目标：让模型提取结构化数值字段，handler 收模型提取结果做确定性比较。对比 regex-only 与 extracted-value 两条路径在黄金样本集上的准确率差异。

## T5.2 目标

不改规则语义。不扩展到 numeric_comparison 以外的规则类型。只给 numeric_comparison 和 sum_numeric_comparison 注入结构化数值提取。对比 regex-only 与 extracted-value 两条路径在黄金样本集上的准确率差异。

## 风险分析（执行前必读）

### 一、按错误类型逐项评估

| 错误类型 | 样本 | 正则失败原因 | 期望 LLM 行为 | 风险等级 | 说明 |
|---------|------|------------|-------------|---------|------|
| 中文数字 | cn-007-POS-01 "二十四个月" | 正则只匹配阿拉伯数字 | 提取 24，单位"月" | 低 | LLM 原生能力。cn-027 验证中已正确处理中文数字表达 |
| 百分之N | cn-006-POS-01 "百分之三" | context_pattern 未命中 | 提取 3，单位"%" | 低 | 简单中文数字转阿拉伯 |
| 万分比 | cn-009-POS-01 "万分之一" | 正则不认识中文分数 | 提取 0.01，单位"%" | 中 | 需要中文分数语义+单位换算。模型可能提取到"万分之一"但不做单位换算，需 prompt 明确指示或后处理 |

### 二、API 依赖风险

DeepSeek Chat API 是外部依赖：
- 单点故障：API 不可用时全路径回退正则
- 速率限制：46 样本 × 每样本一次调用 = 可接受
- 敏感数据：黄金样本是测试数据，无真实合同，无合规风险
- 参考模板：`tests/cn027_road2_verify.py` 已有完整工作模式（Mock pydantic → 环境变量读 key → DeepSeek → 提取 → 注入 validate），可复用

### 三、回归风险（最大风险）

当前 handler 逻辑：structured_input 有值 → 走模型路径；无值或低置信度 null → fallback 到正则。理论上安全，但有一处隐患：

**如果模型提取了错误的值（如 cn-008 中提取了 28 但上下文是结算而非验收），handler 会当正确值做判定，制造新的假阳性或假阴性。**

需要额外加固：模型提取结果 confidence 低于阈值时必须回退正则，而不仅是 null-value 场景回退。建议阈值 ≥ 0.7。

### 四、范围风险（可控）

约束清晰：只动 `tests/` 目录，不动 `app/`、`domains/`。只处理 numeric_comparison 和 sum_numeric_comparison。不改规则语义。LLM 只提取数值不做判定。这些都是硬边界。

## 工作内容

### 第一步：搭建 LLM 提取器

在 `tests/road2_llm_extractor.py` 中实现。

调用 DeepSeek Chat API（OpenAI 兼容接口），只提取结构化字段，不做合规判定。API 配置从环境变量读取：

```
DEEPSEEK_API_KEY  — 密钥
DEEPSEEK_API_URL  — 端点，默认 https://api.deepseek.com/v1
```

可复用 `tests/cn027_road2_verify.py` 的 Mock pydantic 绕过方案。

模型输出 JSON schema：

```json
{
  "field_label": "屋面防水保修期",
  "value": 5,
  "unit": "年",
  "source_text": "屋面防水工程保修期为五年。",
  "confidence": 0.95
}
```

**prompt 设计要求**：
- 明确中文数字→阿拉伯数字转换（"五年"→5，"二十四个月"→24）
- 明确中文分数→小数转换（"万分之一"→0.01，"万分之五"→0.05）
- 只提取与指定 field 上下文相关的数值，不提取无关数字
- 单位保持原单位，不做换算（年→年，月→月，天→天）
- 明确区分"28天提交结算资料"和"28天组织验收"，仅后者对 cn-008 有意义

### 第二步：写对照评测脚本

在 `tests/road2_compare.py` 中实现。

对黄金样本集中每条样本，同时跑两条路径：
- 路径 A（regex）：`kernel.validate(text)` — 现有正则路径
- 路径 B（extracted）：`kernel.validate(text, structured_extractions=llm_result)` — 注入模型提取结果

每个样本输出对照行：

```
sample_id | rule_id | expected | regex_status | extracted_status | extract_value | extract_unit | extract_confidence | error_type
```

脚本运行流程：
1. 加载 `road2_gold_samples.json`
2. 对每个样本调用 LLM 提取器获取结构化字段
3. 将提取结果注入 `structured_extractions` 参数调用 `kernel.validate()`
4. 同时跑正则路径作对照
5. 输出报告到 `tests/road2_compare_report.json`

## 验收标准

1. 黄金样本准确率高于 76.1%（当前正则基线）
2. 假阳性不超过 1 条
3. 每个被模型覆盖的样本保留 evidence：原文片段、提取值、单位、置信度
4. **模型失败时必须回退到正则路径**，不得让已有正确样本退化。confidence < 0.7 的结果应视为不可靠，走正则回退
5. 对照报告中有逐条 delta 分析（哪些样本改善了、哪些未变、哪些退化）
